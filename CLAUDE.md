# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## このリポジトリの目的

中級者向け有料動画教材「**Claude Code入門・実践**」のスライド・台本・実演コードを管理するリポジトリ。ScreenStudioで画面収録して販売する前提。

## 制作フロー

モジュール単位で進行する。各動画は **約5分**。

1. 構成を壁打ち（このリポジトリ内でClaudeと対話）
2. HTMLスライドを生成（`{section}/index.html`）
3. 台本を作成（`{section}/script.md`）
4. Claude Code の実演を検証
5. ScreenStudioで収録

撮影順は **S1→S2→...** の順番。先の回の確定を待ってから次へ進む。

**実演台本を書く前に、必ず動作PoCを通す（必守）**: 「このコマンドで動くはず」で書き始めない。実際に動かして見たフローをそのまま台本にする。概念整理（例: MCPとプラグインの違い、Skillsと◯◯の比較など）は台本には入れない——**「どうすれば動くか」だけ**に絞る。視聴者の混乱を最小化するため。UIのボタン名・設定パス・ショートカットキーも同様に実機で確認してから台本に書く（「設定 → データとプライバシー」「Ctrl+E」のように、存在しない・名前が違うUIを書かないため）。

**SaaSアカウント作成系は「初回ユーザー視点」でPoCする（必守）**: アカウント作成・プロジェクト作成などの初回フローは、既存アカウントで触っているだけだと見落とす画面がある。新規メールアドレスで最初から最後まで通し、通常ユーザーが必ず踏むステップ（組織作成画面、電話番号認証、初回セットアップウィザードなど）を漏らさず台本に入れる。S5V3で「Create a new organization」画面を台本から落とし、SRT検証で発覚した事例あり。

**UIラベルは実機スクリーンショットで都度確認（必守）**: SaaSのUIは半年〜1年で刷新される。「API Docs」「管理画面」「設定ページ」のようなラベルを**記憶や一般知識で書かない**。台本を書く直前に該当画面を開いてスクショを撮り、正確な文言で書く。S5V3で旧Supabaseの「API Docs」を想定して書き、実際は「Get connected → API Keys」の構成だった事例あり。

## スライドの技術構成

**HTMLスライド**（自由度重視、Revealなどのフレームワークは不採用）。
- `styles/slide.css` に共通スタイル。**全モジュールで使い回す**（CSSはここだけに置く）
- 各動画の `index.html` は `../styles/slide.css` を相対パスで読み込む
- 1枚 = `<section class="slide">`。`.slide.active` で表示、JSで切り替え
- 操作: `→` / `Space` / クリックで次、`←` で前、`Home` / `End` で端へ
- `#N` ハッシュでスライド番号指定可能（再収録時の頭出しに便利）

## スライドデザインの統一ルール（必守）

動画越しの視認性が最優先。全モジュールで一貫させる。

- **フォント**: 游ゴシック系（`"游ゴシック", "Yu Gothic", YuGothic, "Noto Sans JP", sans-serif`）
- **フォントサイズ**: **最小18pt**（ナビUIも含む）。本文28pt、見出し44pt、カバー80pt目安
- **配色**: 白基調。文字は `#1a1a1a`、サブ `#555`、ミュート `#888`
- **アクセント**: Claudeオレンジ系 `#c15f3c`（`--color-accent`）
- **文字量**: 1スライド1メッセージ。長文は台本で話し、スライドはキーワードのみ
- **箇条書きは必ず左揃え**: `<ul>` や `<li>` には `text-align:left` を明示する。スライド全体に `text-align:center` がかかるため、指定しないと各行の長さでガタガタになる
- **文言は声に出して読んで確認する**: スライドの見出し・本文を生成したら声に出して読んで自然かチェックする。ぎこちない場合はユーザーに代替案を複数提示してから確定する
- **コードブロック**: `font-family: var(--font-mono)`、サイズも18pt以上を維持
- **カバー**: `h1` のみ。サブコピーは原則入れない（タイトルだけどーんと）。`h1` には必ずその動画の正式タイトルを入れる。スライド内容の要約（例:「モードを使い分けよう」）ではなく、視聴者が何を学ぶかが伝わるタイトル（例:「Claude Code を安全に使う方法を知ろう」）にする
- **次回予告・他動画の参照を入れない**: スライド・台本ともに入れない。撮影順や構成は後から変わるので、個別動画は自己完結させる。「前回のSlack回で〜」「次回のFigmaで〜」のような他動画参照、「SlackとNotionが揃って〜」のような他動画ツールとの対比軸も避ける。1動画1テーマに絞る

CSS変数は `:root` で定義済み。値を変えるときは `styles/slide.css` だけをいじる。

## 構成パターン

- **実演なしの導入回**: S1V1 の11枚パターン（カバー → コースマップ → セクションプレビュー → コアメッセージ → クロージング）
- **実演メインの回**: 4枚で脱力OK（カバー → お題 → プロンプト → クロージング）。実演そのものが主役なのでスライドは額縁だけでよい
- **実演中にスライドを切り替える予定がないなら、そのスライドは作らない**。本番収録後に「出番がなかった」と気付いて削る手戻りは大きい。実演パートが長い回は「スライドで説明する箇所」と「画面だけで進める箇所」の時間配分を先に確定させる（S5V4でPROMPT①/②スライドを作ったが実演ではデスクトップ版画面を直接映して出番なし→7枚→5枚へ削る修正が発生）

## プレビュー

ローカルで `open s1v1/index.html` するだけで開ける（ビルドツール不要）。
収録時はブラウザをフルスクリーン（`F`キー等）にしてScreenStudioでキャプチャ。

## 台本とスライドの対応ルール（必守）

- **スライドを追加・削除したら台本の`[PN]`番号も必ず同時に更新する**。スライド枚数が変わったのに台本番号を放置すると、シーンとセリフが一致しなくなる
- スライドの`<!-- N: タイトル -->`コメントと台本の`## [PN]`は常に1対1で対応させる
- **既存ドラフトを受け取ったらまず整合チェックをする**。セッション開始時に、スライドのコメント番号と台本の`[PN]`番号が1対1で対応しているか確認してから作業に入る
- **スライドのラベル・例文を変更したら台本も全文検索して同期する**。「ペルソナ」「出力言語」のような表現がスライドで変わった場合、台本で同じ言葉に言及している箇所を `grep` で探して更新する。番号の対応だけでなく内容の対応も確認すること
- **台本編集後は `---` 直前の空行を必ず確認する**。`grep -n "^---" script.md` で全箇所チェックし、直前行が空行でない場合は修正してからコミットする

## 台本の書き方

- **ファイル**: `{section}/script.md`
- **文体**: 敬体（ですます調）
- **一人称**: 「私」（「僕」は使わない）
- **構造**: 冒頭に想定尺・実演有無などのメタ情報、以降は `## [PN] タイトル` でスライドと番号同期
- **分量の目安**: 約 **1,600字** ＝ **5.3字/秒 × 300秒**（ゆったりめ。丁寧に読むと5分、普通に読むと4〜4.5分）。実演が長い回は待ち時間があるので、1,200〜1,400字に収めてもよい
- **役割分担**: スライドに書いた情報は読み上げない。削った補足・具体例・感情の動かし方は台本で話す（スライドと**重複させない**）
- **実演パート**: 実演がある動画は `## 【実演パート】タイトル` ブロックを挿入。本文に `（画面を〜に切り替え、〜）` のようなト書きを丸括弧で入れる
- **画面操作の手順は「読み上げる本文」か「黙って操作するト書き」かを明示する**。全角括弧 `（…）` で囲めばト書き（ナレーションしない）、句点 `。` で終える地の文なら読み上げる。同じ画面操作でも演出（黙って操作 vs 手順を口に出しながら操作）で扱いが変わるので、台本の時点で決めておく。例: `（Vercelダッシュボード → Add New → Project → …）` はト書き、`Vercelダッシュボード → Add New → Project → …。` は読み上げ
- **実演パートでは画面で伝わる内容を台本に書かない**: 結果の数値・根拠・画面上の文字列など、視聴者が画面を見れば分かることはナレーションで繰り返さない。台本は画面で伝わらない情報だけに絞る
- **プロンプト読み上げ**: スライドでプロンプト全文を見せる場合、台本にも `>` ブロッククォートで同じ全文を記載する（読み上げやすくするため）
- **プロンプト例はゴール先行で書く**: 「〜を抽出してください」「〜を紐付けてください」のような抽象的な指示より、「来週の経営会議で使う資料を作ってください。〜をHTMLダッシュボードに」のように**何がしたいか（ゴール）を先頭**に書く。Claudeは目的が明確なほど良い出力をする。受講者にとっても「何のためのプロンプトか」が一瞬で伝わる
- **締めの定型**: クロージング末尾に `この動画は以上です。` を入れて明示的に区切る
- **略語は初出時に厳密定義を確認する**: 慣用で使いがちな略語ほど誤用しやすい。台本で略語を出すときは、記憶や「だいたいこう」で書かず、定義を検索してから書く。例: CRUDは4操作（Create/Read/Update/Delete）で、検索はReadの一部として扱う。SPA（単一ページアプリ）、REST、CI/CD、SaaS/PaaS/IaaSなども同様

## ツール名の表記

台本・スライドで混同しないよう以下を使い分ける。

- `Claude.ai` — ブラウザで開くウェブ版
- `Claude Code デスクトップ版` — Mac / Windows のデスクトップアプリ（`Claude Desktop` とは書かない）
- `Claude Code CLI` — ターミナルで `claude` コマンドを実行する動作モード。Claude Code デスクトップ版の内蔵ターミナル／外部ターミナル（Terminal.app等）のどちらでも動く

**対象者の呼び方**: 台本・スライドともに「**経営者**」を使う。「経営層」は使わない。

**スラッシュコマンドの制約（必守）**: `/plugin`・`/mcp`・`/setup` などはすべて `claude` コマンド内専用。Claude Code デスクトップ版の**チャット欄**では認識されない（「ここでは認識されないコマンドです」と表示される）。実演で「設定のため `/plugin` を実行」と書くときは、必ずターミナル側で動作することを台本のト書きで明示する。

## 外部連携の3分類（必守）

Claudeから外部に手を伸ばす手段は3つ。**混同しない／用語を揃える**。

- **Skill** — Claudeに渡す手順書。資格やマニュアルの発想。`.claude/skills/` に置く。S3V1 / V2で扱う
- **MCP** — Model Context Protocolサーバーを追加して機能を拡張する仕組み。公式レジストリ掲載と非公式（コミュニティ製）がある。CLI / デスクトップ版とも利用可、自由度は高いが設定は重め
- **コネクタ** — Anthropic公式が用意した外部サービス連携の窓口。Drive / Gmail / Calendar / Notion / Linear / Figma等が登録済み。Claude Code デスクトップ版の「ディレクトリ → コネクタ」から追加、数クリックでOAuth接続

**コネクタの「接続」と「セッション有効化」は別ステップ**。ディレクトリで追加＋OAuthしただけだとセッションから使えない。入力欄の「+」or「/」メニューの「Connectors」でセッションごとに有効化する必要がある。実演では詰まりポイントなので丁寧に見せる。

教材の役割分担（予定）：
- S3V1 / V2 — Skill
- S3V3 / V4 — MCP
- S3V5 — コネクタ

## Skillの動画を作るとき（必守）

S3V1 / V2のようにSkillを実演する回で守ること。

- **Skillはセッション開始時に読み込まれる**。作成直後の同じセッションでは動かない。一度セッションを終了して新セッションを開く必要がある。MCPと同じ制約。実演で必ず詰まるのでスライドまたはト書きで強調する
- **Skillのトリガーは自然語、スラッシュコマンドではない**。`/skill-name`で呼び出せるわけではなく、descriptionに書いた発動文脈にClaudeが自動反応する。実演では事前に発動フレーズを決めておく（例：「月次レポートを作って」）
- **Skill Creatorのパッケージ名は `skill-creator@claude-plugins-official`**（PoC確認済み）。インストールは `claude` 起動後に `/plugin install skill-creator@claude-plugins-official` を一度実行。その後 `/reload-plugins` でリロードすれば同セッション内で使用可能。起動トリガーは「新しいSkillを作りたい」（自然語）
- **`/plugin list` はCLI専用**。Claude Code デスクトップ版のチャット欄では認識されない。実演でデスクトップ版を使う場合はこのコマンドを台本に入れない

## 実演プロンプトの先回り防御（必守）

PoC省略で本番収録する方針のため、Claudeに渡す実演プロンプトに「**初回実行で詰まるパターン**」の防御を必ず入れる。S6V2でRPC関数の追加実行、S6V3でSQL順序エラー、初期データのリレーション解決で詰まった事例あり。

**SQL生成プロンプトの標準防御文（必ず含める）**:
- 「すべて1つのSQLスクリプトにまとめ、SQL Editorで一発実行で完結する形にしてください。後から追加実行が必要なSQLが残らないようにしてください」
- 「クリーンアップ（DROP）は初回実行（テーブル未作成）でも安全に通るようにしてください。`DROP TRIGGER ... ON テーブル名` は対象テーブルがないとエラーになるので使わず、`DROP TABLE IF EXISTS ... CASCADE` でトリガごと削除する形にしてください」
- 複数テーブルの初期データを入れる場合は「商談のcustomer_idは顧客のIDと正しく紐付けてください」のように**リレーション解決を明示**する

**How to apply**: 新しい実演プロンプトを書くときは、上記の標準防御文をテンプレートとしてコピペし、回ごとの差分だけ調整する。新しいエラーパターンに遭遇したら、ここに防御文を1行追加して次回以降に反映する。

## MCP連携の動画を作るとき（必守）

S3V3 / V4 のようにMCP接続を実演する回で守ること。

- **主体はMCP、プラグインは手段**。タイトル・地の文では「Notion MCP」「Slack MCP」のように**機能側（MCP）**を主役に書く。「プラグイン」「マーケットプレイス」「OAuth」「インテグレーショントークン」等の技術用語は地の文から極力削除。コマンド名（`/plugin install` 等）は実演で打つので残す。「MCPとプラグインの違い」のような概念解説は台本に入れない——受講者を混乱させる
- **`/plugin install` の後は必ず新しいセッションを開き直す**。ClaudeはセッションをロードするタイミングでMCPのツール定義を読み込むため、**同じセッション内ではインストール直後のMCPが反映されない**（`/plugin list` で `connected` 表示でも、会話中のClaudeからは見えない状態になる）。受講者が必ず詰まるポイントなのでスライドで強調する
- **認証の挙動は過去の状態に依存する**。同じワークスペースの認証トークンが残っていれば、再インストールしてもブラウザOAuth画面は開かない。収録時にクリーンな挙動を見せたければ、Notion / Slack 側の連携アプリ管理画面から該当アプリを削除してから撮る

## URL・略称の表記ルール（必守）

- **URLは必ず `https://` から書く**。`claude.ai/design` のような省略形は禁止。正：`https://claude.ai/design`
- **台本にURLを書かない**。読み上げできないため。「こちらのページ」「このリンク」に置き換える
- **略称を使わない**。`PM` は「プロジェクトマネージャー」、`LP` は「ランディングページ」と初出時に必ず展開する

## 台本のMarkdownルール（必守）

- **`---` の直前には必ず空行を入れる**。空行なしで `---` を書くと、直上のテキストがH2見出しとして解釈され、script-viewerの左カラム（シーン名）に誤表示される

## 台本の文体ルール（必守）

- **言い切りで書く**。「〜という事例も報告されています」→「〜です」。「〜というのが感想です」「体感で〜」などの主観表現も禁止
- **強調副詞は原則削除**。「特に」「非常に」「とても」「かなり」などは事実を直接書けば不要。例：「特に得意です」→「得意です」
- **主語と述語を省略しない**。「依頼する」ではなく「作りたいものをClaudeに伝える」のように完全な文にする
- **「おまけ」を使わない**。「さらに」「加えて」に置き換える
- **比喩を使わない**。「司書として動く」「視野に入る」「連鎖がほぐれる」などの詩的表現は、視聴者ごとに解釈が揺れる上に情報量がない。事実を直接書く（例：「データベースを読み込み、経営課題を整理しています」）。ただし、**視聴者の理解を助ける比喩はユーザー確認のうえ特例で許容**（例：MCPとSkillsをキッチンの道具とレシピで説明）
- **「〜についてです」を使わない**。「次に、〜についてです」「〜の話についてです」のような形は不自然な日本語。「〜についてお話します」「〜を確認しておきましょう」のように動詞で締める
- **体言止めを使わない**。「プロジェクト管理データベース。この2つを〜」→「プロジェクト管理データベースです。この2つを〜」。ですます調の統一を崩さない
- **「〇〇するだけ」を使わない**。「伝えるだけ」「渡すだけ」「書くだけ」のような表現は、手軽さをアピールしたいつもりでも内容が浅く見える。行為と結果を具体的に書く（例：「伝えるだけです」→「伝えればClaudeが差分を拾ってくれます」）
- **視聴者が知らない「以前」を比較対象に出さない**。「従来は〜が必要でした」「以前はインテグレーショントークン発行も不要」のような対比は、旧来手順を知らない受講者には意味不明。新しい方法の価値を単独で語る
- **収録者目線でなく受講生目線で書く**。「収録本番では〜」「画面を映さずに〜」のような収録作業目線の表現は禁止。「実際に使うときは〜」のように受講生が自分で試す場面を想定して書く
- **受動的な動詞を避ける**。「Claudeに頼む」「お願いする」「やってもらう」のような受動的表現は、AIに任せきりのイメージを与える。「Claudeと協働で組み上げる」「Claudeと一緒にコードを書く」「指示して動かす」のように、受講者自身が主体で動いている動詞を使う
- **GOALスライドの太字キーワードを台本にそのまま持ち込まない**。スライドの「**Supabase接続**：localStorageを置き換える」をそのまま「**Supabase接続**として、〜置き換えます」と読み上げると、名詞先出しでぎこちなくなる。台本は「まずアプリとSupabaseを接続して、〜置き換えます」のように動詞主体の流れで書く。スライドのキーワードは視覚で伝わるので、台本で再利用しなくてよい

## スライドタイトルのルール（必守）

- **固有名詞をタイトルに入れない**。「Venado LPを作る」→「ランナー向けサンダルのPR案件LPを作る」のように汎用的に
- **英語略称をいきなり出さない**。「Tweaks」「CTA」など初出時は日本語説明をセットで入れる
- **【ターゲット層】の明示は固有名詞ルールの例外**。「【経理・経営企画向け】月次レポート…」のように、業種・職種を【】で括ってカバーに入れるのは許容。ターゲットがズレるリスクを避けるため、複数併記でもよい
- **カバータイトルは最初の試案で確定せず、中身を組んだ後に再考する**。視聴者の最初の3秒を決めるのがカバータイトル。スライド構成・MERITS・実演フローを組んだ後に、「タイトルが動画の主役と一致しているか」を必ず1回見直す。S6V3で「永続化しよう」→「リレーション設計をしよう」に変更した事例あり。中身が固まってから本質を再抽出するのが筋

## スライドのビジュアル検証（必須）

スライドHTMLを編集・作成したら、**必ず**以下を実行して全ページを目視確認する：

```bash
node scripts/screenshot-slides.js <section>/index.html
```

保存先: `/tmp/slide-check/<section>/p1.png` 〜 `pN.png`。`Read` ツールで画像を開いて確認し、以下が問題ないことを検証する：

- テキストの意図しない折り返し・見切れがない
- 文字サイズが十分に大きい（最小18pt）
- 全ページがスライド境界内に収まっている

問題があれば修正してから再度スクリーンショットを撮り直す。ユーザーへの報告前に解決しておくこと。

**3回試行ルール（必守）**: 視聴者の理解に影響しない細部レイアウト（箱の幅統一、列の整列、余白の微調整など）で **3回試行しても改善しなければ諦めて先に進む**。S6V3で「箱の大きさ統一」に4〜5回試行した反省。完璧主義より動画全体の進行を優先する。「自然な見た目」で十分だと判断してユーザー確認に進めた方が早い。
